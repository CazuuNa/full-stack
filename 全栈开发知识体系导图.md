- 结构化日志（含 traceId）、指标（QPS/RT/错误率）、链路追踪

## **3. 多租户模型设计（必须先定）**
你要先决定“租户隔离方式”，建议从最实用的开始：

### **方案 A：共享库共享表 + tenant_id 字段隔离（推荐起步）**
- 所有业务表都带 `tenant_id`
- 所有查询默认自动加 `tenant_id`
- 优点：成本低、好扩展；缺点：必须严防越权查询
- 配套要求（必须做）
  - 后端请求上下文必须拿到 tenantId
  - ORM 层封装“默认加 tenantId”策略
  - 管理员操作需严格鉴权与审计

### **租户识别（tenant resolution）**
- 方式 1：用户登录后 token 内带 tenantId（最常见）
- 方式 2：域名识别（tenantA.xxx.com）+ token 双校验（更 SaaS）
- 方式 3：请求头 X-Tenant-Id（仅内部系统可用，不建议外网）

推荐：**登录时选择租户或根据域名确定租户，token 内写入 tenantId**。

## **4. 权限模型（RBAC 满分版）**
### **角色与权限颗粒度**
- **Permission**：最小权限点，例如 `user.read`、`user.write`、`role.manage`
- **Role**：权限集合
- **User**：用户属于某租户下，可绑定多个角色
- **可选增强**
  - 资源级权限（例如“只能管理自己部门”）→ 后续用 ABAC/数据权限扩展

### **系统内角色建议**
- **平台超管（Platform Admin）**：跨租户管理（只给你自己/运维）
- **租户管理员（Tenant Owner/Admin）**：管理本租户一切
- **普通成员（Member）**：按授权使用

### **权限落点（强烈建议）**
- 后端：以“路由级 + 业务资源级”为主（Guard/Interceptor）
- 前端：菜单与按钮基于 permission 控制（但前端只做展示，安全必须在后端）

## PostgreSQL vs MySQL：哪个更适合你这个 ToB 多租户后台？

结论给你一个“工程向推荐”：
- **优先选 PostgreSQL**：更适合做“规范、复杂查询、约束强、可演进”的 ToB 系统，后期扩展（JSON、复杂报表、权限查询）更舒服。
- **选 MySQL 也完全可以**：生态更广、运维更常见，很多公司默认 MySQL；你如果未来目标公司/团队更偏 MySQL，就选 MySQL。

### 你这个项目的关键对比（面向 ToB 多租户后台）
- **数据一致性与约束**
  - PostgreSQL：约束、外键、事务语义整体更“严谨”，做权限/审计这类强一致业务更省心。
  - MySQL：也成熟，但团队里经常会“为了性能”弱化约束（不一定是坏事，但更考验规范）。
- **复杂查询与可表达性**
  - PostgreSQL：CTE、窗口函数、复杂聚合、JSON 查询能力普遍更强，做后台筛选/报表常用。
  - MySQL：也支持不少高级特性，但整体上复杂分析类 SQL 体验通常不如 PG。
- **JSON/半结构化数据**
  - PostgreSQL：JSONB 很强，适合存租户配置、动态表单配置等（当然仍建议关键数据结构化建表）。
  - MySQL：JSON 也有，但生态与能力通常略弱一档。
- **团队与环境因素（最重要）**
  - 如果你未来工作环境大概率是 MySQL（很多传统业务/中小公司）：选 MySQL 学习收益更直接。
  - 如果你希望把项目做得更“工程化、规范化、能力上限高”：选 PostgreSQL 更合适。

我的建议：你做这个“满分项目”就用 **PostgreSQL**，并在文档里写清楚选择理由；面试时也更好讲“为什么这样选”。

## “租户识别”是什么意思？

多租户系统里，请求进来必须先回答一个问题：**这个请求属于哪个企业/哪个租户（tenant）？**  
这一步就叫 **租户识别（Tenant Resolution）**。识别出来的 `tenantId` 会被写入“请求上下文”，之后所有数据库读写默认都要带上 `tenant_id` 过滤，防止串数据。

---

## 两种租户识别方式有什么不同？

你前面看到的两种常见方式本质区别是：**租户信息从哪里来、对用户体验和安全边界有什么影响**。

### 方式 A：登录后“选择租户”/ Token 内携带 tenantId（最常见、最适合起步）
- **流程**
  - 用户登录 → 系统返回该用户可加入的租户列表 → 选择一个租户 → 颁发包含 `tenantId` 的 token
- **优点**
  - 实现简单、清晰
  - 适合一个账号属于多个租户的场景（顾问/外包/集团员工）
- **缺点**
  - 前端需要一个“选择租户”步骤（如果用户多租户归属较多会频繁选择）
- **适用**
  - 大多数 ToB SaaS、内部系统、后台管理系统

### 方式 B：子域名/域名识别（tenantA.yourapp.com）（更 SaaS、更“满分体验”）
- **流程**
  - 用户访问 `tenantA.xxx.com` → 系统根据域名定位 tenant → 登录后 token 仍携带 tenantId（通常还会校验域名与 tenantId 匹配）
- **优点**
  - 用户体验更好：天然“进入某个企业空间”
  - 品牌感强：每个租户一个入口
- **缺点**
  - 需要域名与证书、Nginx/网关配置更复杂
  - 本地开发/测试环境要做 hosts 或通配域名（例如 *.localtest.me）
- **适用**
  - 真正对外 SaaS 产品、希望每个租户独立入口的场景

Vite 和 Next.js 的核心区别不是“谁更快”，而是**定位不同**：Vite 是“前端构建工具/开发服务器”，Next.js 是“带后端能力的 React 全栈框架”。

---

## **1）定位差异**
- **Vite**
  - 你得到的是一个 React SPA（单页应用）工程模板
  - 路由、鉴权、数据请求、SEO 等都要你自己选库并组织
  - 主要负责：开发构建速度、打包、HMR
- **Next.js**
  - 你得到的是“框架级”能力：路由、SSR/SSG、API Route（可选）、中间件等
  - 可以做同构渲染（服务端渲染），更适合需要 SEO/首屏性能的应用

---

## **2）渲染与 SEO**
- **Vite（典型 SPA）**
  - 默认 CSR（浏览器渲染）
  - SEO 一般（需要额外方案，如预渲染/SSR 自己搭）
- **Next.js**
  - 原生支持 SSR/SSG/ISR
  - SEO、首屏性能天然更强

---

## **3）路由与工程约束**
- **Vite**
  - 通常配 `react-router-dom`，路由完全自主
  - 自由度大，但规范需要你自己定（目录、加载、鉴权守卫）
- **Next.js**
  - 文件路由（App Router / Pages Router）
  - 约束更强，很多最佳实践框架已定好

---

## **4）“是否需要 Node 服务”**
- **Vite**
  - 纯前端站点可以直接静态部署（CDN/Nginx）
  - 你的后端（NestJS）是独立服务
- **Next.js**
  - 即使只做前端，也通常需要一个 Node 运行环境（SSR 时必需）
  - 可以把 BFF/API 层做在 Next（但你这里已经有 NestJS，一般不必把逻辑放两边）

---

- 不太依赖 SEO
- 更看重：权限、表格表单、工程化、接口规范、部署简单

**User**

创建pnpm-workspace.yaml得作用是什么。

**Assistant**

`pnpm-workspace.yaml` 的作用是把你的仓库声明成一个 **pnpm 的 Workspace（单仓多包/monorepo）**，让 pnpm 知道“哪些目录是子项目（package）”，并据此提供一整套跨包管理能力。

**它解决的核心问题**
- **统一依赖安装**：你在仓库根目录执行一次 `pnpm install`，pnpm 会给 workspace 里的所有包安装依赖，而不是每个目录单独装一遍。
- **本地包互相引用（workspace 协议）**：比如 `apps/web` 依赖 `packages/shared`，可以用 workspace 方式链接（开发时直接引用本地源码/构建产物），不需要发布到 npm。
- **一键批量执行命令**：支持 `pnpm -r dev`、`pnpm -r build` 这种递归执行，在多个子包里同时或按依赖顺序执行脚本。
- **依赖去重与硬链接加速**：pnpm 本身的 store + workspace 能让多包共享同一份依赖，减少磁盘占用、加快安装。
- **约束工作区范围**：通过 `packages: - "apps/*"` 这类配置，明确哪些目录算 workspace 包，避免把无关目录（docs、infra）误当成包。

**没有它会怎样**
- pnpm 不会把你的仓库当 workspace 管理：`pnpm -r ...` 不好用或无效
- 跨包引用 `packages/shared` 会变麻烦（要手动 link、或用相对路径、或发布私包）
- 安装与脚本执行更碎片化，工程体验明显下降

一句话：`pnpm-workspace.yaml` 就是“告诉 pnpm：这是一个 monorepo，以及 monorepo 里有哪些包”。
